# /debug — 系統性除錯

**核心原則：** 找到根因後才嘗試修復。症狀修復等於失敗。

## 鐵律

```
沒有完成根因調查就不能提出修復
```

如果 Phase 1 未完成，你不能提出修復方案。

## 相關 Skills

- **testing** — 在實作修復前寫失敗測試（Phase 4）
- **/fix** — 確認根因後的完整 TDD 修復流程

---

## Phase 1：根因調查

**完成所有步驟後才能繼續。**

1. **仔細閱讀錯誤訊息** — stack trace、行號、錯誤碼、Inner Exception
2. **穩定重現** — 能否可靠觸發？如果不行，收集更多資料
3. **檢查近期變更** — `git log`、`git diff`、新依賴、設定變更
4. **追蹤資料流** — 錯誤值從哪裡產生？從 call stack 向上追蹤
5. **多層系統加診斷** — 在 Controller → Service → Repository 各邊界加 log，執行一次看哪裡斷開
6. **檢查三層架構** — 問題是否源於跨層的不正確呼叫？

## Phase 2：模式分析

1. 在同一個 codebase 找到類似功能的正常範例
2. 比較正常 vs 異常 — 列出每一個差異（不管多小）
3. 理解異常程式碼的依賴和假設

## Phase 3：假設與測試

1. 陳述一個假設：「我認為 X 是根因，因為 Y」
2. 做最小的變更來測試假設 — 一次只改一個變數
3. 成功 → Phase 4。失敗 → 提出新假設。**不要疊加修復**

## Phase 4：修復

1. 寫一個重現根因的失敗測試（使用 **testing** skill）
2. 實作一個針對根因的修復
3. 驗證：測試通過、無回歸

**如果已經嘗試 3 次以上修復都失敗：** 停下來。問題可能出在架構 — 先討論再嘗試更多修復。

---

## 紅旗 — 回到 Phase 1

| 想法 | 現實 |
|------|------|
| 「先快速修一下，之後再調查」 | 補丁掩蓋根因 |
| 「試試 X 看能不能用」 | 盲猜浪費時間 |
| 「我不完全理解但這可能有用」 | 不完整的理解必然產生新 bug |
| 「再試一次修復」（已失敗 2 次以上） | 3 次以上失敗 = 架構問題 |
| 在追蹤資料流前就提出方案 | Phase 1 還沒完成 |
